# Surprise文章ルール

AI-Driven Schoolの運営会社である株式会社Surpriseが、普段Slackやドキュメントを書くときに使っている**社内の文章ルール**です。

第9回の月次課題で配布された「提案スキル」を自分用にカスタマイズするときの、参考のひとつとして配布します。

---

## ひとことで言うと

- 件名と冒頭で**結論を先に**書く。読み手が3秒で内容を判断できる状態を目指す
- 1文は短く、**情報は「■→◎→・」の階層**で整理する
- **事実・解釈・提案を分け**、選択肢を出すときは必ず推奨案を理由つきで示す
- 期限・数値・抽象語は**具体的に**書く（「なるはや」ではなく日時、「効率化」ではなく何をするか）

迷ったら「相手が一度読んだだけで、同じ情景が頭に浮かぶか」を基準に判断してください。

---

## なぜ文章ルールがあるのか

**文章ルールがあるのは、書く力がそのまま「考える力」と「人を動かす力」になるからです。**

文章を書くことは考えることそのものです。
頭の中で「考えた」つもりでも、文字にしなければ考えは整理されません。
書いて初めて、次の2つがそろいます。

■ 論理が整う
事実と理由が並び、筋が通っているかを確かめられる

■ 気持ちが整う
自分が本当に伝えたいことと、相手の受け取り方が見えてくる

この2つがそろうと、正しく決められ、相手に動いてもらえます。
逆に、伝わらない文章は、書いた本人がまだ考えきれていない文章です。

だからこのルールは、守るべき作法ではありません。
考えを整理し、相手に最短で届けるための技術です。
以下のすべてのルールは、「論理を整える」「気持ちを整える」の双方を助けます。

文章を読むのは、たいてい自分より忙しい相手です。
わかりにくい文章は、相手の時間と「判断する力」を奪います。
理想は、相手が1行読んだだけで「それでOK」と先に進める状態です。
リンクを開かせたり、長文を最後まで読ませたりするのは、その対極にあります。

## 1. 件名と冒頭で要点を明確に伝える

**ルール**
- 件名には【共有】【提案】【依頼】【質問】【相談】のタグを付け、動詞または体言止め（名詞で文末を止める表現）で終える
- 本文の冒頭で、結論や要点を簡潔に書く

**理由**：読み手が一目で内容を把握でき、すぐ判断・行動できる。

**具体例**

悪い例:

```
明日の打ち合わせについて

参加予定のメンバーの多くが別件と重なっているようで、山田さんからは13時からは参加できないと連絡がありました。鈴木さんも終日予定が入っていて...（以下長文）
```

良い例:

```
【相談】明日の会議時間🕐

明日の会議を15時に変更したいです。

■ 理由
◎ 参加者の都合
山田さんは13時以降なら参加可能です。
鈴木さんは午後のみ参加可能です。
◎ 設備の都合
会議室は15時以降なら確保可能です。
```

## 2. 「〜について」「〜の件」を使わない

**ルール**：件名やタイトルに「〜について」「〜の件」などの曖昧な表現を使わない。

**理由**：「〜について」は逃げの言葉です。何を決めたいのか、相手にどう動いてほしいのかを見極める仕事を放棄して、それっぽくまとめてしまっています。件名で「相談・共有・報告・提案・依頼」のどれなのかをはっきりさせると、相手は開く前に内容を予測でき、すぐ対応できます。

**具体例**

悪い例:

```
【共有】予算について
```

良い例:

```
【共有】2025年度マーケティング予算を10%削減します
```

## 3. 文章は短く区切る

**ルール**：1文は30字以内に収める。長い文は短く区切る。

**理由**：短い文は理解しやすく、特にチャットでは重要。

**具体例**

悪い例:

```
昨日開催された販売戦略会議で議論されたマーケティング予算について、各部署から集まった意見をまとめると今年度は前年比10%増で進めることになりました。
```

良い例:

```
昨日の販売戦略会議でマーケティング予算を議論しました。
各部署の意見をまとめた結果、今年度は前年比10%増で進めます。
```

## 4. 長文には概要を添える

**ルール**：本文が6行以上になる場合、最初の3行以内に全体の要点をまとめる。

**理由**：読み手が内容の全体像をつかみやすくなる。

**具体例**

悪い例:

```
【共有】新プロジェクト調査結果の報告

まず市場規模を調べ直したところ、当初の想定を上回っていました。
競合の動きも気になり各社を確認したところ、3社が類似サービスを出していました。
今後は来週あたりに会議を開いて方針を決めたいと考えていて...（と詳細が続く）
```
（最後まで読まないと全体像がわからない）

良い例:

```
【共有】新プロジェクト調査結果の報告

■ 概要
市場規模は予想の2倍です。
競合3社が類似サービスを展開中です。
来週中に戦略会議を実施します。

詳細は以下の通りです。
まず市場規模については...（以下詳細）
```

## 5. 情報を階層構造で整理する

**ルール**：情報の重要度に応じて「■」「◎」「・」の階層を作り、主要ポイントと詳細を区別する。

**理由**：情報の関係性がわかりやすくなり、読み手が整理しやすくなる。

**具体例**

悪い例:

```
【共有】歓迎会の準備状況
・準備70%完了
・お店の予約済み
・参加者15人確定
・プレゼント未購入
・日程6/20(金)
```
（主要ポイントと細目が同じ並びで、どれが重要かが見えない）

良い例:

```
【共有】歓迎会の準備状況

■ 準備70%完了
◎ 完了したこと
・お店の予約：6/20(金)19時、個室
・参加者：15名確定
◎ 残っていること
プレゼント購入
◎ 当日の予定
6/20(金)19時から始めます。
```

## 6. 箇条書きは漏れなく重複なく（MECE）

**ルール**：箇条書きの項目は網羅的で重複がなく（MECE＝漏れなくダブりなく）、語尾を統一する。

**理由**：情報が整理され、抜け漏れや混乱を防げる。

**具体例**

悪い例:

```
■ 来期の投資先
・広告費
・マーケティング費
・人材採用
```
（「広告費」と「マーケティング費」が重なり（ダブり）、システム投資などにも触れていない（漏れ）。これでは決裁者が判断できない）

良い例:

```
■ 来期の投資先（合計1000万円）
・広告：500万円
・人材採用：300万円
・システム改修：200万円
```
（重複なく全体を覆い、合計も合うので、決裁者が見た瞬間に判断できる）

## 7. 期限は日時まで具体的に指定する

**ルール**：期限は「YYYY/MM/DD HH:MM」の形式で時間まで明確に指定する。「なるはや」は使わない。

**理由**：曖昧さをなくし、期限の認識ズレを防ぐ。

**具体例**

悪い例:

```
なるべく早く送ってください
```

良い例:

```
2025/05/20 15:00までに送ってください
```

## 8. 自分の意見・推奨を明示する

**ルール**：「推奨：○○」「推し案：○○」の形で自分の意見を明確に示す。

**理由**：事実を整理するだけの文章より、「私はこうしたい」という提案のほうが価値があります。提案を避けてしまうのは、①間違ったら恥ずかしいという怖さ、②どれが良いか考えるのが面倒、という2つの理由からです。それでも提案を書くほど、自分で考え抜く量が増え、仕事の腕（市場価値）が上がります。「どうしますか？」ではなく「私はAが良いと思います。理由は〜」と書きましょう。

**具体例**

悪い例:

```
A案とB案があります。どちらにしますか？
```

良い例:

```
■ 選択肢
◎ A案：コスト重視
◎ B案：品質重視
→推し案：A案（理由：コスト20%削減可能）
```

## 9. 事実・解釈・提案を分ける

**ルール**：客観的な事実、自分の解釈、行動の提案を、明確に区別して伝える。

**理由**：論理の流れが明確になり、説得力が増す。

**具体例**

悪い例:

```
【提案】ウェブサイト改善策の実施💻

CVRが2.1%から1.5%に落ちたのは、たぶん先月の試作で導線が複雑になったせいだと思うので、デザインを簡素化してABテストもやったほうがいいと思います。
```
（確かな事実と、推測と、提案が1文に混ざり、どこまでが事実か判断できない）

良い例:

```
【提案】ウェブサイト改善策の実施💻

■ 事実
CVR 2.1%→1.5%に低下

■ 解釈
先月の試作でランディングページの導線が複雑になり、CVRが低下した可能性がある

■ 提案
・デザイン簡素化
・ABテスト実施
```

## 10. 外部リンクだけに頼らない

**ルール**：重要な情報や数値は本文に直接書き、リンクは補足として扱う。

**理由**：リンクを開かなくても、読めば内容を理解できるようにする。

**具体例**

悪い例:

```
■ 4月売上実績
4月の売上をまとめました。詳細はこちら [リンク]
```
（リンクを開かないと数字が一切わからない）

良い例:

```
■ 4月売上実績
4月の総売上は1240万円（前年比+17%）でした。
主力商品Aが855万円、商品Bが385万円です。

詳細データはこちら [リンク]
```

## 11. 懸念と対策を示す

**ルール**：提案や計画には、想定される懸念と対応策を含める。

**理由**：先回りして不安を解消し、スムーズな合意を得やすくなる。

**具体例**

悪い例:

```
【提案】新サービスの来月リリース

明日から準備を開始し、来月1日にリリースします。
```
（懸念に触れていないため、読み手は不安が残り、すぐに合意できない）

良い例:

```
【提案】新サービスの来月リリース

明日から新サービスの準備を開始し、来月1日にリリースします。

■ 懸念と対策
◎ 想定される懸念
アクセス急増によるサーバー負荷が予想されます。初日は通常の3倍のアクセスを見込んでいます。

◎ 対策内容
事前に自動スケーリング設定を完了させます。監視体制を強化して即時対応できるようにします。
```

## 12. 選択肢を網羅して推奨案を示す

**ルール**：意思決定が必要な場合、主要な選択肢をすべて示したうえで推奨案を提示する。

**理由**：比較検討の材料が揃い、より良い判断ができる。選りすぐりの案だけを出すと、読み手が判断できない。

**具体例**

悪い例:

```
今回は4週間・中コストの案で進めたいです。バランスが良いからです。
```
（どんな選択肢を比べた結果かが見えず、本当にそれが最適か判断できない）

良い例:

```
■ 選択肢
◎ プランA
短納期で開発できますが、コストは高めです。最短3週間で完了します。

◎ プランB
品質重視で進めるため納期は長くなりますが、コストは抑えられます。およそ6週間かかります。

◎ プランC
バランス型で、中程度の納期とコストです。4週間程度で完了見込みです。

→推奨はプランCです。品質とスピードのバランスが最適だと判断しました。
```

## 13. 全体から詳細へ書く

**ルール**：最初に全体像を示してから、詳細に入る。

**理由**：起きたことを順番に並べていく「小学生の作文」は、忙しい相手には読まれません。結論を最後に置くと、相手はそこまで読まないと要点がわからないからです。結論を先に書けば、相手は「それで？（で、何をすればいい？）」の一言で仕事を進められます。

**具体例**

悪い例（小学生作文：起きたことを時系列で並べ、本題が最後に来る）:

```
先週木曜にチームで集まり、タスク管理の方法を議論しました。
その結果、ツールを導入するのが有効だという意見が出ました。
いくつか調査し、機能や価格を比較し、最終的に2つに絞りました。
それぞれの長所と短所を比べた結果、ツールAを導入したいと考えました。
導入してよいか相談したいです。
```
（最後の1行まで読まないと「何を相談したいのか」がわからない）

良い例（結論を先に置く）:

```
【相談】タスク管理ツールAを導入したいです

先日の会議で、チームにタスク管理ツールを入れることが決まりました。
候補を比較した結果、価格と使いやすさで優れたツールAを推します。
導入してよいか、ご確認をお願いします。
```

## 14. 正しい階層で箇条書きする

**ルール**：複数項目を列挙する場合は適切な階層（■ → ◎ → ・）を使い、同じ階層には同レベルの情報だけを入れる。

**理由**：情報の関係性が明確になり、読みやすくなる。

**具体例**

悪い例:

```
■ プロジェクト状況
・進捗50%
・課題あり
・担当：佐藤
・来週テスト
```

良い例:

```
■ プロジェクト状況
◎ 進捗状況
全体の50%が完了しました。デザインは完了し、現在コーディング中です。

◎ 課題
リソース不足で遅延の懸念があります。特にテスト工程が未計画です。

◎ 今後の予定
来週からテストを開始します。再来週の完了を目指します。
```

## 15. 箇条書きの語尾を統一する

**ルール**：同じ階層の箇条書きは、語尾を名詞止めか動詞止めのどちらかに統一する。

**理由**：読みやすさが向上し、情報が整理される。

**具体例**

悪い例:

```
◎ 主要施策
・顧客満足度を向上させる
・コスト削減
・ブランドの強化を図る
```

良い例:

```
◎ 主要施策
・顧客満足度の向上
・コストの削減
・ブランド力の強化
```

## 16. 敬語は簡潔にする

**ルール**：過剰な敬語表現を避け、丁寧でありながら簡潔な表現を使う。

**理由**：過剰に丁寧な文章は、丁寧どころか卑屈で堅い印象を与えます。書く側も読む側も時間がかかり、双方が損をします。スッと頭に入る短い文章を書くほうが、よほど相手への気づかいになります。「ございます」「申し上げます」を「です・ます」に直すだけで、読む速さが変わります。

**具体例**

悪い例:

```
お手数をおかけしまして誠に恐縮ではございますが、ご確認いただけますでしょうか
```

良い例:

```
ご確認ください
```

## 17. 抽象的な言葉は定義する

**ルール**：「品質」「早急に」「効率化」など、解釈が分かれる言葉には具体的な定義を添える。

**理由**：抽象的な言葉は、相手から「何も言っていない」「何も考えていない」と受け取られます。いくらでも解釈できる言葉は、結局だれも動けないからです。具体的になっているかは、次の3つで確かめられます。

① あとで達成できたかどうかを確認できる
② 数字が入っている
③ だれが担当するか（責任者）がはっきりしている

**具体例**

悪い例:

```
作業を効率化します
```

良い例:

```
作業時間を◯月◯日12:00までに、現状の半分に短縮します
```

## 18. 意味の区切りで改行する

**ルール**
・文と文の間（句点「。」の後）で改行する
・1文が短い場合（30字以内）は無理に改行しない
・文の途中では改行しない

**理由**：内容の区切りが視覚的に明確になり、理解しやすくなる。文の途中で改行すると、PCで読みやすくしたつもりでも、スマホでは不自然な位置で折り返されてかえって読みにくくなる。

**具体例**

悪い例:

```
先日ご依頼いただいた件について調査結果をご報告いたします。スケジュール遅延の主な原因は仕様変更が頻繁に発生したこととテスト段階でリソースが不足したことの2点です。
```
（改行がなく、どこで内容が区切れるのかが目で追えない）

良い例:

```
先日ご依頼いただいた件について、調査結果をご報告します。

スケジュール遅延の主な原因は、以下の2点です。
・仕様変更が頻繁に発生したこと
・テスト段階でリソースが不足したこと
```

## 19. 短い文章は階層化しない

**ルール**：200字以内かつ要点が1つの場合は、階層化せず簡潔な文章で伝える。

**理由**：シンプルな内容は、構造化よりも端的な表現の方が伝わる。

**具体例**

悪い例:

```
【共有】明日のミーティングを明後日に変更します🕐

■ 変更内容
◎ 変更理由
都合が合わない方が多かったため変更しました。
```
（要点が1つしかないのに階層を作り、かえって読みにくい）

良い例:

```
【共有】明日のミーティングを明後日に変更します🕐

都合が合わない方が多かったため変更しました。
```

## 20. 番号リストは順序・手順のみに使う

**ルール**：「1. 2. 3.」の番号付きリストは、順序や手順を示す場合のみ使う。順番に意味がないときは、番号を安易に使わず「・」で並べる。

**理由**：番号の意味が明確になり、情報が整理される。

**具体例**

悪い例:

```
■ 改善点
1. コスト削減
2. 売上向上
3. ブランド強化
```
（順序や優先度がないのに番号を振ると、1番が最重要だと誤解される。この場合は「・」を使う）

良い例:

```
■ 実施手順
1. 要件定義を完了する
2. デザイン案を作成する
3. クライアントの承認を得る
```

## 21. 具体例は引用・コードブロックで示す

**ルール**：具体例や補足説明は「例：」や引用記号（>）を使い、本文と区別する。

**理由**：本文と例が明確に区別され、理解しやすくなる。

**具体例**

悪い例:

```
顧客対応マニュアルを更新しました。例えばクレーム対応は初回10分以内に電話連絡するといった内容です。
```
（どこからが具体例なのかが本文に溶け込み、区別がつかない）

良い例:

```
顧客対応マニュアルを更新しました。

> 例：クレーム対応は初回10分以内に電話連絡
```

## 22. 対比は構造で表現する

**ルール**：「しかし」「ですが」などの逆接表現は使わず、項目を分けて構造的に対比を示す。

**理由**：逆接を使うと前置きが長くなります。「順調なんですが、今週の会議は延期したいです」のように、一番言いたいこと（延期したい）が後ろに回ってしまうからです。逆接が出てきたら「前置きが長くなっている」というサインだと考え、一番言いたいことを先に置きましょう。

**具体例**

悪い例:

```
このプランはコストが低いですが、懸念が大きいです。
```

良い例:

```
■ このプランの評価
◎ メリット
低コストで実施可能です。前回比30%削減できます。

◎ デメリット
開発期間短縮による懸念があります。特に品質担保が課題です。
```

## 23. 順接の「が」で文をつなげない

**ルール**：順接の「が」（「〜ですが、…」と話をただ続けるだけの「が」）を使わず、文を分ける。

**理由**：順接の「が」は前置きを長くし、一番言いたいことを後ろに追いやります。「順調ですが、今週の会議は延期したいです」のように、結論（延期したい）が後回しになるからです。「が」が出てきたら文を区切り、言いたいことを先に置きましょう。

**具体例**

悪い例:

```
資料は完成していますが、共有は明日にさせてください。
```

良い例:

```
資料は完成しています。共有は明日にさせてください。
```

## 24. 「また」「さらに」を減らす

**ルール**：「また」「さらに」「加えて」「そのうえ」などの接続詞を減らし、箇条書きや段落で整理する。

**理由**：これらの接続詞を重ねると、その場の思いつきで適当に書いたように見えます。情報がどんどん足されていくと、結局何が言いたいのかわからなくなるからです。代わりに「ポイントは3つあります」と先に数を示し、箇条書きで並べると、相手は受け取る準備ができます（ルール34を参照）。

**具体例**

悪い例:

```
新機能を追加しました。また、バグも修正しました。さらに、UIも改善しています。
```

良い例:

```
■ アップデート内容
◎ 機能面の改善
新機能を追加しました。これによりデータ分析が容易になります。

◎ 不具合対応
複数のバグを修正しました。特に検索機能の不具合が解消されています。

◎ 使い勝手の向上
UIを改善しました。ボタン配置を直感的に変更しています。
```

## 25. コロンは見出し用途のみで使う

**ルール**：コロン（：）（:）は見出しや定義の用途のみに使い、箇条書きの中では使わない。

**理由**：視認性が高まり、読点との混在による混乱を防げる。

**具体例**

悪い例:

```
■ 目標
・売上: 10%増加
・施策: 広告予算拡大
```

良い例:

```
■ 目標
売上10%増加

■ 施策
・広告予算を拡大します。
・SNS活用を強化します。
```

## 26. 単独の箇条書き記号は使わない

**ルール**：下位項目が1つしかない場合は箇条書き記号（◎・など）を使わず、上位階層に含めて表現する。

**理由**：無駄な階層が減り、構造がシンプルになる。

**具体例**

悪い例:

```
■ 今後の予定
・明日会議
```

良い例:

```
■ 今後の予定
明日会議を開催します。13時から会議室Aで行います。
```

## 27. 絵文字でビジュアルをわかりやすくする

**ルール**：業務連絡では、件名や強調したいポイントに絵文字を活用する。ただし多用は逆効果。

**理由**：適切な絵文字は視認性を高める。

**活用例**

```
⏰ 期限・締切
⚠️ 重要事項
👉 推奨事項
✅ 完了項目
🔗 参照リンク
```

## 28. 同じ階層の箇条書きは抽象度を揃える

**ルール**：同じ階層の箇条書きは抽象度（情報の粒度）を揃え、内容とフォーマットを統一する。

**理由**：情報の整理がしやすく、読みやすい文章になる。

**具体例**

悪い例:

```
■ 会議の準備
・資料を印刷する
・会議を成功させる
・お茶を用意する
```
（「会議を成功させる」は目標であって作業ではない。具体的な準備と並べると粒度がバラバラ）

良い例:

```
■ 会議の準備
・資料を印刷する
・お茶を用意する
・プロジェクターをつなぐ
```

## 29. 同種情報が3つ以上続くなら箇条書きにする

**ルール**：同じ種類の情報や例を3つ以上列挙する場合は、通常の文章ではなく箇条書き形式を使う。

**理由**：情報の比較や把握が容易になり、視覚的に整理される。

**具体例**

悪い例:

```
バーカウンターでスタッフが食材やビニール袋を置いていて、お客様や受講生の目の前でネギや袋が並んでいて、さらに袋を開けるビニール音も顧客体験を妨げていました。
```
（同じ種類の指摘が3つあるのに1文に詰め込まれ、いくつ問題があるのか把握しづらい）

良い例:

```
◎ バーカウンターの不適切な使用
・スタッフが食材やビニール袋をカウンターに置いていました
・お客様や受講生の目の前で、ネギや袋が並んでいました
・袋を開ける音（ビニール音）も顧客体験を妨げていました
```

## 30. 予定は日時指定で確定表現にする

**ルール**：「〜する予定」ではなく「〜します」と確定的に表現し、日時まで明示する。

**理由**：仕事とは、自分が引き受けたことに責任を持つことです。「予定」ではなく日時で「します」と言い切ると、その責任を引き受ける姿勢が伝わり、相手も安心して任せられます。

**具体例**

悪い例:

```
レポートは来週中に提出する予定です
```

良い例:

```
レポートを5/24(金) 17:00に提出します
```

## 31. 用語は一貫して使う

**ルール**：同じ意味のものを、別の表現で言い換えない。一度使った言葉は同じ意味で使い続け、別の言葉に置き換えない。特に名詞・専門用語・プロジェクト名の表記揺れに注意する。

**理由**：表現の揺らぎは読み手に無用な混乱を与え、誤解を生む。用語を統一すると、文章全体の明確性が高まる。

**具体例**

悪い例:

```
クライアントからの要望をヒアリングし、顧客満足度向上の施策を検討しました。お客様の声を反映します。
```
（「クライアント」「顧客」「お客様」が混在）

良い例:

```
クライアントからの要望をヒアリングし、クライアント満足度向上の施策を検討しました。クライアントの声を反映します。
```
（「クライアント」に統一）

## 32. 名前は「〜さん」づけと敬語を基本にする

**ルール**：名前を呼ぶときは「〜さん」づけと敬語を基本とする（お互いに信頼関係が構築できている場合は除く）。

**理由**
・呼び捨て・君づけ・あだ名・タメ口は、無意識に上から目線の態度につながりやすい
・特に学生や若いメンバーに対して気をつける
・ただし社内で過剰な謙譲語・尊敬語はやりすぎなので使わない
・社外（お客様・取引先）に対しては、特に丁寧な言葉づかいを徹底する。いじるような言葉遣いはしない
・Slackのメンション（`@名前`）では「さん」を付けない（`@makonari` のように呼ぶ）

## 33. 「おつかれさまです」等の前置きは省く

**ルール**：「おつかれさまです」「〜さん」などの前置きは省略し、要件を簡潔に投稿する。

**理由**：メールではないので、形式張った前置きは不要。

## 34. 列挙の前に項目数を伝える

**ルール**：同じ種類の情報を並べるときは、列挙する前に「◯つあります」と数を先に伝える。

**理由**：先に数がわかると、相手は「この話は3つ覚えればいい」と受け取る準備ができます。「また」「さらに」で情報を足し続けるより、何個の話なのかが最初に見えるからです。伝えたいことが本当は何個あるかではなく、相手が覚えられるように「3つに絞って伝える」と決めるのがコツです（ルール24・ルール29と合わせて使う）。

**具体例**

悪い例:

```
好きな食べ物は麻婆豆腐です。さらに二郎系ラーメンも好きです。加えてグリーンカレーも好きです。
```

良い例:

```
好きな食べ物は3つあります。
・麻婆豆腐
・二郎系ラーメン
・グリーンカレー
```
